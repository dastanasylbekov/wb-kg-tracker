# Трекер изменений справочного центра Wildberries

Отслеживает дату «Обновлено DD.MM.YYYY» у статей справочного центра WB для продавцов
по нескольким странам и показывает, **у каких статей изменилась дата** (значит, внесли
правки), какие статьи **появились** и какие **исчезли**. Работает сам по будням через
GitHub Actions — сервер не нужен.

## Что где смотреть

| Где | Что показывает |
|---|---|
| `STATUS.md` (в корне) | Таблица «пульс»: когда был последний прогон каждой страны, сколько статей, сколько изменений. Быстрый признак, что всё живо. |
| `history.html` (через GitHub Pages) | Наглядная страница: переключатель стран + лента изменений за 90 дней. Тянет данные сама. |
| `data/{страна}/CHANGES.md` | Что изменилось в стране за **последний** прогон. Перезаписывается каждый раз. |
| `data/{страна}/HISTORY.md` | Лента изменений страны **за 90 дней**. Появляется, когда есть первое изменение. |

## Как это устроено

1. **`wb_kg_tracker.py`** — обходит раздел `/instructions/{язык}/{страна}/` выбранной страны
   (categories → category → subcategory → material), у каждой статьи берёт заголовок, дату
   «Обновлено» и признак «для этой страны». Сайт отдаётся сервером готовым (SSR), поэтому
   headless-браузер не нужен — хватает `requests`.
2. Результат по каждой стране пишется в свою папку **`data/{страна}/`**, чтобы страны не
   смешивались. Файлы: `snapshot.json` (текущее состояние — нужно для сравнения на следующем
   прогоне), `changes_history.json` (накопленная история), `CHANGES.md`, `HISTORY.md`,
   `last_run.json`.
3. **`.github/workflows/track-wb-kg.yml`** по будням проходит по списку стран, запускает
   скрипт для каждой, затем собирает `STATUS.md` и одним коммитом сохраняет всё в репозиторий.
4. **`history.html`** — статичная страница: список стран зашит в ней, данные она подгружает
   из `data/{страна}/changes_history.json` через `fetch`. Ничего пересобирать не нужно —
   обновились данные в репозитории, обновилась и страница.

## Структура репозитория

```
wb_kg_tracker.py                     # скрипт-обходчик
history.html                         # дашборд (статичный, тянет данные сам)
STATUS.md                            # пульс: авто-генерируется прогоном
.github/workflows/track-wb-kg.yml    # расписание и запуск
data/
  kg/  by/  kz/  am/  uz/  tj/        # по папке на страну
  ge/                                 # Грузия (русская версия)
  ge-ka/                              # Грузия (грузинская версия)
    snapshot.json  changes_history.json  CHANGES.md  HISTORY.md  last_run.json
```

## Запуск

- **По расписанию:** будни в 03:00 UTC (09:00 по Бишкеку) — cron в workflow.
- **Вручную:** вкладка Actions → workflow → Run workflow.
- **Локально:** `pip install requests beautifulsoup4 lxml`, затем
  `python wb_kg_tracker.py --country kg` (одна страна за вызов; `--status` собирает STATUS.md).

## Какие страны отслеживаются

Список задаётся **в двух местах** (при изменении правьте оба):

1. `.github/workflows/track-wb-kg.yml`, строка `COUNTRIES` — что собирает скрипт.
   Формат: код страны, либо `код:язык` (напр. `ge:ka` — Грузия на грузинском).
   Сейчас: `kg by kz am uz tj ge ge:ka`.
2. `history.html`, массив `COUNTRIES` в начале `<script>` — что показывает дашборд.

Коды: `kg` Кыргызстан, `by` Беларусь, `kz` Казахстан, `am` Армения, `uz` Узбекистан,
`tj` Таджикистан, `ge` Грузия, `ru` (общая российская справка — большая, по умолчанию не
включена).

### Как добавить страну

1. В `wb_kg_tracker.py` при необходимости дополните справочники в начале файла:
   `COUNTRY_NAMES`, `COUNTRY_FLAGS`, и `COUNTRY_MARKERS` (фраза «для продавцов из …»).
2. Допишите код в `COUNTRIES` в workflow.
3. Допишите страну в массив `COUNTRIES` в `history.html`.

## Особые случаи (заложены в коде)

- **Грузия — на своём домене.** У Грузии справка на `seller.wildberries.ge`, а не `.ru`.
  Это учтено в словаре `COUNTRY_BASE`. Для новой страны на отдельном домене — добавьте туда.
- **Таджикистан — без маркера.** На страницах TJ нет фразы «для продавцов из …», поэтому
  все статьи раздела считаются местными (страна в наборе `NO_MARKER_COUNTRIES`).
- **Грузинский язык.** Для `ge:ka` дата ищется по грузинской метке `განახლდა` (в словаре
  `UPDATED_LABELS`) плюс запасной поиск первой даты на странице. Если у `ge-ka` даты выходят
  пустыми — проверьте реальное слово-метку на грузинской странице и поправьте `UPDATED_LABELS`.

## Дашборд и GitHub Pages

`history.html` подгружает данные по относительным путям (`data/…`), поэтому работает,
когда открыт **через GitHub Pages**, а репозиторий **публичный** (у приватного нет
публичного URL для `fetch`).

Включение: Settings → Pages → Source = Deploy from a branch → Branch `main`, папка `/ (root)`
→ Save. Через 1–3 минуты появится ссылка. Дашборд открывается по адресу вида
`https://<логин>.github.io/<репозиторий>/history.html`.

Примечания: у Pages есть кэш CDN (~10 минут), поэтому свежие данные на странице могут
появиться с небольшой задержкой после прогона. Для публичных репозиториев минуты GitHub
Actions не ограничены.

## Уведомления в Telegram (необязательно)

Если добавить два секрета (Settings → Secrets and variables → Actions), при изменениях бот
пришлёт сводку. Без них всё работает, просто без уведомлений.

- `TG_TOKEN` — токен бота от @BotFather.
- `TG_CHAT` — ваш chat id (можно узнать через @getidsbot).

## Обслуживание и типичные проблемы

- **У страны 0 статей в `STATUS.md`.** Обход не нашёл ссылки — возможно, у раздела другая
  разметка или страна на отдельном домене. Проверьте URL `/instructions/ru/{код}/categories`
  в браузере.
- **«Дата не распознана у N статей» в логе.** WB, возможно, изменил вёрстку или слово-метку.
  Дата ищется регуляркой `DATE_RE` (и `UPDATED_LABELS` для других языков) — поправьте её.
- **Прогон падает на коммите.** Нужны права на запись: Settings → Actions → General →
  Workflow permissions → Read and write permissions.
- **История обнулилась.** Так бывает, только если удалить `data/{страна}/snapshot.json`
  (тогда все статьи снова «новые») или `changes_history.json` (стирается лента). Эти файлы
  не трогаем.

## Лимиты

Прогон ~30 минут по будням ≈ 700 минут/месяц. Бесплатный план приватного репозитория даёт
2000 Linux-минут/месяц; публичного — без ограничений. Лимит трат по умолчанию $0, поэтому
неожиданных списаний не будет.

## Зависимости

`requests`, `beautifulsoup4`, `lxml` (устанавливаются в workflow автоматически).
